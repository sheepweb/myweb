# 节点健康检查设置说明

## 概述

节点健康检查用于自动检测节点的可用性和延迟，帮助管理员及时发现故障节点。系统会定期检查所有节点，标记离线或延迟过高的节点。

## 配置位置

- **管理员后台** → **系统设置** → **节点健康检查**

## 一、检查间隔（分钟）

### 1. 检查间隔设置

- **默认值**：30 分钟
- **说明**：系统每隔多长时间检查一次所有节点的健康状态

**工作原理：**
- 系统会按设定间隔自动检查所有节点
- 检查所有启用的节点（`IsActive = true`）
- 更新节点的状态（online/offline）和延迟信息

**建议值：**
- **频繁检查**：15-30 分钟（适合对可用性要求高的场景）
- **标准检查**：30-60 分钟（推荐，平衡检查频率和服务器负载）
- **低频检查**：60-120 分钟（适合节点稳定、检查频率要求不高的场景）

**注意事项：**
- 检查频率过高会增加服务器负载
- 检查频率过低可能无法及时发现故障
- 建议根据节点数量和服务器性能调整

## 二、最大允许延迟（毫秒）

### 1. 延迟阈值设置

- **默认值**：3000 毫秒（3 秒）
- **说明**：节点延迟超过此值时，会被标记为超时或离线

**工作原理：**
- 系统测试节点连接时，记录响应时间
- 如果延迟超过设定值，节点可能被标记为超时
- 超时的节点可能不会在订阅中返回给用户

**建议值：**
- **严格**：1000-2000 毫秒（适合对速度要求高的场景）
- **标准**：3000 毫秒（推荐，平衡速度和可用性）
- **宽松**：5000 毫秒或更长（适合网络环境较差的场景）

**注意事项：**
- 延迟阈值过低可能导致正常节点被误判为超时
- 延迟阈值过高可能影响用户体验
- 建议根据实际网络环境调整

## 三、测试超时时间（秒）

### 1. 单节点超时设置

- **默认值**：5 秒
- **说明**：单个节点测试的最大等待时间，超过此时间仍未响应则判定为超时

**工作原理：**
- 系统测试每个节点时，会等待此时间
- 如果在此时间内没有响应，判定为超时
- 超时的节点会被标记为离线或超时状态

**建议值：**
- **快速**：3-5 秒（适合节点数量多、需要快速检查的场景）
- **标准**：5-10 秒（推荐，平衡检查速度和准确性）
- **宽松**：10-15 秒（适合网络环境较差的场景）

**注意事项：**
- 超时时间过短可能导致正常节点被误判
- 超时时间过长会增加检查总耗时
- 建议根据节点数量和网络环境调整

## 四、测速网站 URL

### 1. 测速地址配置

- **默认值**：`https://ping.pe`
- **说明**：用于测试节点延迟的网页地址

**工作原理：**
- 如果配置了测速网站 URL，系统会通过 HTTP 请求测试节点延迟
- 如果留空，系统使用 TCP 直接连接测试
- HTTP 测试可以更准确地反映实际使用体验

**推荐地址：**
- `https://ping.pe`（推荐，稳定可靠）
- `https://www.google.com`（备用）
- `https://www.baidu.com`（国内服务器可用）

**留空说明：**
- 如果留空，系统使用 TCP 直接连接测试
- TCP 测试更快，但可能不如 HTTP 测试准确
- 建议配置测速网站 URL，获得更准确的延迟信息

## 五、配置示例

### 示例 1：标准配置（推荐）

```
检查间隔：30 分钟
最大允许延迟：3000 毫秒
测试超时时间：5 秒
测速网站 URL：https://ping.pe
```

**适用场景：** 大多数场景，平衡检查频率、准确性和服务器负载。

### 示例 2：高频检查配置

```
检查间隔：15 分钟
最大允许延迟：2000 毫秒
测试超时时间：3 秒
测速网站 URL：https://ping.pe
```

**适用场景：** 对节点可用性要求极高，需要快速发现故障。

### 示例 3：低频检查配置

```
检查间隔：60 分钟
最大允许延迟：5000 毫秒
测试超时时间：10 秒
测速网站 URL：（留空，使用 TCP 测试）
```

**适用场景：** 节点稳定，检查频率要求不高，减少服务器负载。

## 六、节点状态说明

### 1. 节点状态类型

- **online**：在线，延迟正常
- **offline**：离线，无法连接
- **timeout**：超时，延迟过高或响应超时

### 2. 状态更新

- **自动更新**：健康检查会自动更新节点状态
- **实时更新**：管理员手动测试节点时，状态立即更新
- **延迟信息**：系统会记录每个节点的延迟信息

## 七、注意事项

1. **服务器负载**：检查频率过高会增加服务器负载，建议根据实际情况调整
2. **网络环境**：测速网站 URL 必须可从服务器访问，否则测试会失败
3. **节点数量**：节点数量多时，建议适当延长检查间隔，避免检查耗时过长
4. **延迟阈值**：根据实际网络环境调整延迟阈值，避免误判

## 八、常见问题

### Q1: 节点检查失败？

**检查项：**
- 测速网站 URL 是否可访问
- 服务器网络是否正常
- 节点服务器是否正常运行
- 查看系统日志中的错误信息

### Q2: 节点被误判为离线？

**可能原因：**
- 延迟阈值设置过低
- 测试超时时间设置过短
- 网络环境不稳定

**解决方法：**
- 适当提高延迟阈值
- 延长测试超时时间
- 检查网络连接

### Q3: 检查耗时过长？

**可能原因：**
- 节点数量过多
- 检查间隔设置过短
- 测试超时时间设置过长

**解决方法：**
- 适当延长检查间隔
- 缩短测试超时时间
- 考虑分批检查节点

### Q4: 如何手动测试节点？

**方法：**
- 在 **节点管理** → **节点列表** 中，点击「测试节点」按钮
- 或使用「批量测试」功能测试多个节点
- 测试结果会立即更新节点状态
