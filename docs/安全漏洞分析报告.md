### 🟡 中危漏洞

#### 1. 错误信息可能泄露敏感信息
- **位置**: `internal/middleware/logger.go:51`
- **问题**: 生产环境可能返回详细错误信息
- **风险**: 可能泄露系统内部信息
- **修复建议**: 确保生产环境只返回通用错误信息

#### 2. 前端console.log可能泄露信息
- **位置**: 多个前端文件
- **问题**: 开发环境的console.log可能在生产环境暴露
- **风险**: 可能泄露调试信息
- **修复建议**: 使用环境变量控制，生产环境禁用

#### 3. 文件上传验证可以加强 ✅ 已修复
- **位置**: `internal/api/handlers/config.go:320-374`
- **问题**: 
  - 文件类型验证基于扩展名（可被绕过）
  - 缺少文件内容验证（MIME类型检查）
- **风险**: 可能上传恶意文件
- **修复建议**: 
  - 添加文件内容验证（读取文件头）
  - 限制文件执行权限
  - 扫描上传文件

#### 4. SQL查询中的LIKE注入风险 ✅ 已修复
- **位置**: `internal/api/handlers/admin.go:29, 97, 100`
- **问题**: 使用 `LIKE '%"+userQuery+"%'` 虽然使用了参数，但LIKE模式可能有风险
- **风险**: 特殊字符可能导致查询问题
- **修复建议**: 对LIKE参数进行转义

```go
// 当前代码
query = query.Where("user_id IN (SELECT id FROM users WHERE username LIKE ? OR email LIKE ?)", 
    "%"+userQuery+"%", "%"+userQuery+"%")
```

- **位置**: `internal/utils/errors.go:44-67`

- **风险**: 日志可能泄露用户信息

## 🔍 详细分析

### 1. CSRF中间件问题

**问题代码**:
```go
if token == "" || !manager.ValidateToken(sessionID, token) {
    origin := c.GetHeader("Origin")
    referer := c.GetHeader("Referer")
    host := c.Request.Host
    if (origin != "" && !isValidOrigin(origin, host)) || (referer != "" && !isValidReferer(referer, host)) {
        // ... 这里缺少大括号闭合
```

**修复方案**:
```go
if token == "" || !manager.ValidateToken(sessionID, token) {
    origin := c.GetHeader("Origin")
    referer := c.GetHeader("Referer")
    host := c.Request.Host
    if (origin != "" && !isValidOrigin(origin, host)) || (referer != "" && !isValidReferer(referer, host)) {
        utils.ErrorResponse(c, http.StatusForbidden, "CSRF验证失败：无效的请求来源", nil)
        c.Abort()
        return
    }
    
    newToken, _ := manager.GenerateToken(sessionID)
    isSecure := c.Request.TLS != nil || c.GetHeader("X-Forwarded-Proto") == "https"
    c.SetCookie("csrf_token", newToken, 86400, "/", "", isSecure, false)
    c.Header("X-CSRF-Token", newToken)
    utils.ErrorResponse(c, http.StatusForbidden, "CSRF验证失败，请刷新页面后重试", nil)
    c.Abort()
    return
}
```
