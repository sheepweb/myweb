- **缓存时间**：24小时
- **缓存键**：`geoip:{ip}`
- **命中率**：80-90%
- **性能提升**：200-500ms → 10-50ms

### 2. 订阅配置缓存
- **缓存内容**：完整的订阅配置（Base64/Clash）
- **缓存时间**：智能TTL（1-10分钟）
- **缓存键**：`subscription:config:{url}:{format}`
- **命中率**：90-95%
- **性能提升**：300-1200ms → 5-20ms

### 3. 节点列表缓存
- **缓存内容**：系统节点列表
- **缓存时间**：1小时
- **缓存键**：`nodes:system:all`
- **命中率**：接近100%
- **性能提升**：100-500ms → 5-20ms

## 二、可以添加的缓存 🔥

### 1. 用户信息缓存（高优先级）⭐⭐⭐⭐⭐
**问题**：每次请求都查询用户信息
```go
// 当前：每次都查询数据库
var user models.User
db.First(&user, userID)
```

**优化方案**：
- 缓存键：`user:info:{user_id}`
- 缓存时间：5-10分钟
- 失效条件：用户信息变更时清除
- 预期提升：20-50ms → 1-5ms

**适用场景**：
- 每个 API 请求都需要用户信息
- 用户信息变化不频繁
- 高频访问

### 2. 用户套餐信息缓存（高优先级）⭐⭐⭐⭐⭐
**问题**：每次都查询用户等级、套餐信息
```go
// 当前：每次都 JOIN 查询
db.Preload("UserLevel").First(&user, userID)
```

**优化方案**：
- 缓存键：`user:level:{user_id}`
- 缓存时间：10分钟
- 失效条件：购买套餐、升级时清除
- 预期提升：30-80ms → 1-5ms

### 3. 订阅信息缓存（高优先级）⭐⭐⭐⭐⭐

### 4. 套餐列表缓存（中优先级）⭐⭐⭐⭐

### 6. 公告列表缓存（中优先级）⭐⭐⭐
**问题**：公告列表频繁查询

**优化方案**：

### 7. 统计数据缓存（低优先级）⭐⭐⭐
**问题**：仪表盘统计数据计算慢

**优化方案**：

- 失效条件：定时刷新

**优化方案**：

## 五、缓存实施原则

### 1. 缓存时间设计
- **频繁变化**：1-5分钟（订阅状态、用户信息）
- **偶尔变化**：10-30分钟（套餐、公告）
- **很少变化**：1小时以上（系统配置、节点）

### 2. 缓存失效策略
- **主动失效**：数据变更时立即清除缓存
- **被动失效**：TTL 到期自动失效
- **强制刷新**：提供手动刷新接口

### 3. 缓存键命名规范
```
{模块}:{类型}:{标识}

例如：
user:info:123          # 用户信息
subscription:config:abc123  # 订阅配置
packages:list:active   # 活跃套餐列表
stats:dashboard:2026-03-05  # 统计数据
```

### 4. 降级策略
- Redis 不可用时自动降级
- 缓存异常时清除并重新查询
- 记录缓存命中率监控
